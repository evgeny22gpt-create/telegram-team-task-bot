# Telegram-бот задач на Python, aiogram, SQLite и Google Sheets

## Назначение документа

Этот материал предназначен для загрузки в NotebookLM. Он описывает устройство реального учебного проекта `Telegram Team Tasks Bot`, объясняет принятые решения, термины, поток данных, безопасность, тестирование и порядок выполнения домашней работы.

Репозиторий проекта: https://github.com/evgeny22gpt-create/telegram-team-task-bot

## Краткое резюме

Проект представляет собой Telegram-бота для хранения командных задач. Пользователь взаимодействует с ботом через команды. Бот сохраняет данные в SQLite, показывает список задач, формирует CSV для Excel и синхронизирует задачи с Google Таблицей.

Основной стек:

- Python 3.14;
- aiogram 3.29.1;
- SQLite3;
- gspread 6.2.1;
- Google Sheets API;
- Google Apps Script;
- unittest;
- Git и GitHub.

Ключевые свойства проекта:

- токен и закрытый ключ не хранятся в коде;
- данные SQLite сохраняются после перезапуска;
- задачи выводятся в порядке добавления;
- CSV корректно открывается в Excel;
- синхронизация сначала строит dry-run;
- конфликты не перезаписываются автоматически;
- применение синхронизации доступно только владельцу;
- перед изменениями создаётся резервная копия SQLite;
- проект имеет 43 автоматических теста и evidence ручных проверок.

## 3. Архитектура проекта

### 3.1. Разделение ответственности

`bot.py`

- принимает события Telegram;
- связывает команды с обработчиками;
- создаёт ответы;
- передаёт работу базе, CSV и синхронизации.

`config.py`

- загружает переменные процесса;
- использует локальный `.env` как fallback;
- проверяет обязательный токен;
- строит абсолютные пути;
- преобразует owner IDs в числа;
- возвращает объект `Settings`.

`database.py`

- создаёт таблицу SQLite;
- мигрирует старую схему;
- добавляет и читает задачи;
- выполняет UPSERT при синхронизации;
- создаёт резервную копию через SQLite backup API.

`exporter.py`

- формирует CSV;
- добавляет UTF-8 BOM;
- защищает Excel от интерпретации пользовательского текста как формулы.

`google_sync.py`

- проверяет заголовки и строки Google;
- преобразует строки в `Task`;
- строит план действий;
- определяет конфликты;
- применяет безопасные действия.

`google_apps_script/Code.gs`

- реагирует на ручное редактирование;
- обновляет `Обновлено (UTC)`;
- не затрагивает заголовок и другие листы.

`tools/`

- содержит отдельные диагностические команды;
- позволяет проверять Google, SQLite, CSV и Telegram identity вне polling.

`tests/`

- проверяет функции без рабочего токена;
- использует временные базы и фальшивый Google Worksheet.

### 3.2. Поток данных

Добавление задачи:

1. Пользователь отправляет `/add Подготовить отчёт`.
2. aiogram создаёт объект `Message` и `CommandObject`.
3. Обработчик проверяет наличие аргумента.
4. Username превращается в подпись автора.
5. `add_task()` валидирует текст.
6. SQLite выполняет параметризованный INSERT.
7. Новый `id` возвращается обработчику.
8. Бот сообщает номер сохранённой задачи.

CSV:

1. `/list_csv` читает `get_all_tasks()`.
2. `build_tasks_csv()` пишет заголовки и строки в память.
3. Результат кодируется как UTF-8 с BOM.
4. `BufferedInputFile` отправляет байты пользователю.

Google sync:

1. `/sync` загружает настройки.
2. gspread авторизуется через сервисный аккаунт.
3. Google-лист читается в режиме `spreadsheets.readonly`.
4. SQLite и Google преобразуются в два списка `Task`.
5. `build_sync_plan()` создаёт решения по каждому `id`.
6. Бот показывает количества, но ничего не записывает.
7. `/sync_apply confirm` проверяет owner ID.
8. Актуальный план строится повторно.
9. Конфликт блокирует применение.
10. Перед изменениями создаётся backup SQLite.
11. Разрешённые изменения применяются.

## 5. config.py подробно

### 5.1. Почему настройки вынесены отдельно

Если читать `os.environ` во всех файлах, появляются дублирование и разные правила. `config.py` формирует единый контракт.

```python
@dataclass(frozen=True, slots=True)
class Settings:
    bot_token: str
    db_path: Path
    google_spreadsheet_id: str
    google_sheet_name: str
    google_service_account_file: Path
    telegram_owner_ids: frozenset[int]
```

`frozen=True` запрещает случайно менять настройки после загрузки.

`slots=True` фиксирует набор полей и уменьшает вероятность опечатки в имени атрибута.

### 5.2. Приоритет переменных

Сначала используется системная переменная процесса. Локальный `.env` применяется только если такой переменной ещё нет.

Для этого используется:

```python
os.environ.setdefault(key, value)
```

Это позволяет на сервере задавать секреты средствами платформы, не переписывая файл.

### 5.3. Стабильные пути

```python
PROJECT_DIR = Path(__file__).resolve().parent
```

Относительный `tasks.db` превращается в путь внутри проекта. Текущая директория терминала перестаёт влиять на расположение базы.

### 5.4. Возможности config.py

- обязательные и необязательные настройки;
- значения по умолчанию;
- удаление парных кавычек;
- поддержка абсолютных и относительных путей;
- проверка положительных Telegram ID;
- свойство `google_sync_configured`;
- ранняя понятная ошибка при отсутствии токена.

## 7. SQLite подробно

### 7.1. Схема

```sql
CREATE TABLE IF NOT EXISTS tasks (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    text TEXT NOT NULL,
    user TEXT NOT NULL,
    created_at TEXT NOT NULL,
    status TEXT NOT NULL DEFAULT 'Новая',
    category TEXT NOT NULL DEFAULT 'Без категории',
    updated_at TEXT NOT NULL DEFAULT ''
)
```

Назначение полей:

- `id` — устойчивый идентификатор и порядок;
- `text` — задача;
- `user` — автор;
- `created_at` — время создания;
- `status` — состояние задачи;
- `category` — произвольная группа;
- `updated_at` — время последнего изменения для синхронизации.

### 7.2. Параметризованный запрос

Неправильно:

```python
query = f"INSERT INTO tasks (text) VALUES ('{text}')"
```

Правильно:

```python
connection.execute(
    "INSERT INTO tasks (text, user, created_at) VALUES (?, ?, ?)",
    (text, user, timestamp),
)
```

Параметры предотвращают SQL-инъекцию и корректно обрабатывают кавычки.

### 7.3. Транзакция

При синхронизации используется UPSERT нескольких задач в одной транзакции.

Если возникает исключение:

```python
connection.rollback()
```

Если всё успешно:

```python
connection.commit()
```

### 7.4. Миграция

Проект проверяет `PRAGMA table_info(tasks)` и добавляет отсутствующие столбцы через `ALTER TABLE`.

Для старых задач:

- `status = Новая`;
- `category = Без категории`;
- `updated_at = created_at`.

Данные до миграции сохраняются в backup, а целостность проверяется.

### 7.5. Backup API

Простое копирование файла работающей SQLite потенциально опасно. Используется встроенный метод:

```python
source.backup(destination)
```

После него:

```sql
PRAGMA integrity_check
```

Ожидаемый результат: `ok`.

## 8. aiogram подробно

### 8.1. Router

```python
router = Router(name="team_tasks")
```

Router хранит правила обработки событий.

### 8.2. Обработчик

```python
@router.message(Command("list"))
async def handle_list(message: Message, db_path: Path) -> None:
    tasks = get_all_tasks(db_path)
    await message.answer(build_tasks_list_text(tasks))
```

### 8.3. Dependency injection

```python
dispatcher["db_path"] = settings.db_path
dispatcher["settings"] = settings
```

Aiogram сопоставляет имя параметра обработчика с объектом в контексте Dispatcher.

### 8.4. Polling

```python
await dispatcher.start_polling(bot)
```

Polling долго работает и получает обновления Telegram.

### 8.5. Блокирующая библиотека

gspread выполняет синхронные HTTP-запросы. Если вызвать его прямо внутри async-обработчика, event loop будет ждать.

Решение:

```python
report = await asyncio.to_thread(run_google_sync_preview, settings)
```

## 10. CSV подробно

CSV состоит из колонок:

```text
ID, Задача, Пользователь, Создано (UTC), Статус, Категория
```

Проблемы совместимости:

1. Excel может неправильно определить UTF-8.
2. Запятые и кавычки внутри текста требуют экранирования.
3. Значение может быть воспринято как формула.
4. Username с `@` требует аккуратной обработки.

Решения:

- использовать стандартный модуль `csv`;
- кодировать как `utf-8-sig`;
- нейтрализовать formula-like text;
- проверять реальный файл в Excel.

Ручной эксперимент подтвердил:

- русский текст отображается;
- username не имеет лишнего видимого апострофа;
- статус и категория можно изменить;
- порядок строк сохраняется.

## 12. Google Apps Script

Простой триггер:

```javascript
function onEdit(e) {
  if (!e || !e.range) return;

  const sheet = e.range.getSheet();
  if (sheet.getName() !== 'Задачи') return;

  // Проверка колонок B, E, F и запись времени в G.
}
```

Важные свойства:

- запускается только от ручного редактирования;
- API-запись сама не вызывает `onEdit`;
- `setValues` внутри onEdit не создаёт рекурсию;
- Deploy не требуется;
- заголовок не изменяется;
- вставка нескольких строк поддерживается.

Проверка выполнялась временной сменой категории и возвратом исходного значения. Колонка G получила новое ISO 8601 время.

## 13. Алгоритм синхронизации

### 13.1. Модель задачи

```python
@dataclass(frozen=True, slots=True)
class Task:
    id: int
    text: str
    user: str
    created_at: str
    status: str
    category: str
    updated_at: str
```

### 13.2. Действия

`PUSH_TO_GOOGLE`

- SQLite является источником новой версии.

`PULL_TO_SQLITE`

- Google является источником новой версии.

`NO_CHANGE`

- содержимое совпадает.

`CONFLICT`

- содержимое различается;
- `updated_at` одинаковый;
- нельзя автоматически определить победителя.

### 13.3. Почему конфликт блокирует весь apply

Частичное применение усложняет понимание состояния. Безопаснее сначала решить конфликт, затем повторить план.

### 13.4. Порядок применения

1. Считать оба источника.
2. Валидировать заголовки, ID, статусы и даты.
3. Построить решения.
4. Если есть conflict — остановиться.
5. Если изменений нет — завершить no-op.
6. Создать SQLite backup.
7. Применить PULL одной транзакцией SQLite.
8. Выполнить Google updates и append rows.
9. Повторить dry-run и получить NO_CHANGE.

## 17. Типичные ошибки и причины

### ModuleNotFoundError: aiogram

Вероятная причина: используется Python другого проекта.

Проверка:

```powershell
Get-Command python
.\.venv\Scripts\python.exe -c "import aiogram; print(aiogram.__version__)"
```

### Бот не отвечает

Проверить:

- процесс запущен;
- `getMe` возвращает правильный username;
- нет второго polling процесса с тем же токеном;
- stderr log не содержит exception.

### CSV отображает кракозябры

Использовать UTF-8 BOM и повторно открыть файл.

### SpreadsheetNotFound

Проверить:

- Spreadsheet ID;
- доступ сервисного аккаунта;
- точное имя листа;
- правильный JSON key.

### Google изменение не попадает в SQLite

Проверить:

- сработал ли onEdit;
- обновилась ли колонка G;
- является ли время новее;
- что показывает `/sync`;
- нет ли conflict.

### Apply ничего не изменил

Если источники совпадают, это корректный no-op. Backup при отсутствии изменений не нужен.

## 18. Глоссарий

**API token** — секрет, позволяющий программе управлять ботом.

**Polling** — постоянное получение новых событий Telegram.

**Router** — набор правил «какое событие вызывает какую функцию».

**Dispatcher** — центральный объект обработки событий aiogram.

**Handler** — функция, отвечающая на событие.

**Dependency injection** — передача зависимостей обработчику по параметрам.

**SQLite** — встраиваемая реляционная база в одном файле.

**Transaction** — группа операций «всё или ничего».

**UPSERT** — INSERT новой строки или UPDATE существующей.

**Migration** — изменение схемы базы без потери данных.

**CSV** — текстовая таблица с разделителями.

**BOM** — служебная последовательность, помогающая определить UTF-8.

**Service Account** — техническая Google-учётная запись программы.

**Scope** — набор разрешений OAuth.

**Dry-run** — расчёт результата без изменения состояния.

**Apply** — фактическое выполнение подготовленного плана.

**Conflict** — различие, для которого нельзя безопасно выбрать источник.

**ISO 8601** — однозначный текстовый формат даты и времени.

**UTC** — всемирное координированное время.

**Git commit** — зафиксированное состояние файлов.

**.gitignore** — правила исключения локальных файлов из Git.

## 19. Контрольные вопросы с краткими ответами

### Почему config.py полезнее прямого os.getenv в каждом модуле?

Он создаёт единый контракт, валидирует значения и предотвращает дублирование.

### Почему SQLite подходит для домашней работы?

Не требует отдельного сервера, но позволяет изучить SQL, схему и транзакции.

### Почему задачи сортируются по id?

AUTOINCREMENT ID отражает порядок добавления.

### Почему created_at хранится в UTC?

UTC не зависит от локальной зоны. Для показа время преобразуется в МСК.

### Почему updated_at нужен отдельно?

created_at не меняется, а updated_at позволяет сравнивать версии.

### Почему конфликт нельзя молча решить в пользу Google?

SQLite могла быть изменена независимо. Молчаливый выбор может потерять данные.

### Почему /sync read-only?

Диагностика должна быть безопасной и доступной без риска записи.

### Почему /sync_apply требует owner ID и confirm?

Публичный username бота позволяет написать ему любому человеку. Двойная проверка защищает изменение данных.

### Почему backup создаётся только при наличии изменений?

No-op не меняет базу, поэтому новая копия создаёт лишь шум.

### Почему Apps Script не требует Deploy?

Функция с зарезервированным именем `onEdit` является простым триггером привязанной таблицы.

### Почему API update не запускает onEdit?

Простые триггеры реагируют на ручное действие пользователя, а не на программное изменение.

### Почему тесты используют FakeWorksheet?

Чтобы проверить применение плана без сетевых запросов и изменения рабочей таблицы.

### Почему Git email заменён на noreply?

Чтобы личный адрес не публиковался в истории коммитов.
