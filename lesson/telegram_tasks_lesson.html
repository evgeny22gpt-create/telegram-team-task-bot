ГЛАВА 01

Не «скрипт», а маленькая система

Минимальная версия домашнего задания состоит из четырёх команд и одной таблицы SQLite. Мы идём дальше и сразу учимся разделять ответственность между файлами.

Интерфейс

Telegram принимает команды и показывает человеку результат.

Логика

Python решает, что сохранить, выгрузить или синхронизировать.

Данные

SQLite хранит задачи на диске, а Google Таблица даёт удобное командное представление.

Обязательный результат

КомандаЧто должен сделать бот
/startПоздороваться и показать справку.
/add ТекстПроверить текст и сохранить новую задачу.
/listПоказать задачи по порядку добавления.
/list_csvСформировать CSV в памяти и отправить документом.
Главная идея урокаКаждая новая возможность должна иметь понятное место в архитектуре и проверку, которая доказывает её работу.
ГЛАВА 02

Карта проекта

Когда всё находится в одном файле, первая версия кажется проще. Но уже на CSV и Google API становится трудно понимать, где искать ошибку.

telegram-team-task-bot/
├── bot.py                  # Telegram и aiogram
├── config.py               # настройки и секреты
├── database.py             # SQLite
├── exporter.py             # CSV
├── google_sync.py          # правила синхронизации
├── google_apps_script/     # onEdit для Google Таблицы
├── tools/                  # диагностика и ручной запуск
├── tests/                  # автоматические проверки
├── evidence/               # скриншоты и протоколы
└── requirements.txt

Связность ниже

CSV ничего не знает о Telegram. База ничего не знает о Google. Их соединяет верхний уровень.

Ошибки локализуются

Проблема SQL ищется в database.py, а ошибка текста ответа — в bot.py.

ГЛАВА 03

Изолированное окружение

Виртуальное окружение — отдельная коробка с библиотеками конкретного проекта. Оно защищает проекты друг от друга.

py -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install -r requirements.txt
Типичная ошибкаКоманда python может указывать на интерпретатор другого проекта. Надёжная диагностика: Get-Command python и python --version.
Зачем фиксировать версии в requirements.txt?

Чтобы на другом компьютере установился тот же интерфейс библиотек. В проекте зафиксированы aiogram==3.29.1 и gspread==6.2.1.

ГЛАВА 04

config.py — диспетчер настроек

config.py не «запускает бота». Он собирает настройки из окружения, превращает строки в удобные типы и останавливает запуск, если обязательных данных нет.

@dataclass(frozen=True, slots=True)
class Settings:
    bot_token: str
    db_path: Path
    google_spreadsheet_id: str
    google_sheet_name: str
    google_service_account_file: Path
    telegram_owner_ids: frozenset[int]

Что это даёт

  • Одна точка входа. Другие модули получают готовый объект Settings.
  • Стабильные пути. База и ключ ищутся относительно папки проекта, а не случайной папки терминала.
  • Приоритет окружения. Системная переменная сильнее локального .env.
  • Проверка типов. Telegram ID превращаются в множество положительных чисел.
Секреты не являются кодомТокен, JSON-ключ и owner ID хранятся только в .env и credentials/. В Git попадает лишь безопасный .env.example.
ГЛАВА 05

SQLite: память бота на диске

SQLite — это база данных внутри одного файла. Сервер не требуется, но SQL, транзакции и схема остаются настоящими.

CREATE TABLE IF NOT EXISTS tasks (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    text TEXT NOT NULL,
    user TEXT NOT NULL,
    created_at TEXT NOT NULL,
    status TEXT NOT NULL DEFAULT 'Новая',
    category TEXT NOT NULL DEFAULT 'Без категории',
    updated_at TEXT NOT NULL
)

Почему данные переживают перезапуск?

Процесс Python завершается, но файл tasks.db остаётся. При следующем запуске CREATE TABLE IF NOT EXISTS не удаляет существующую таблицу.

Почему в SQL стоят знаки вопроса?

connection.execute(
    "INSERT INTO tasks (text, user, created_at) VALUES (?, ?, ?)",
    (clean_text, clean_user, timestamp),
)

Параметры отделяют данные пользователя от текста SQL-команды. Это защищает от SQL-инъекций и проблем с кавычками.

Миграция без потери данныхПроект умеет открыть старую таблицу из четырёх колонок и добавить status, category, updated_at через ALTER TABLE.
ГЛАВА 06

aiogram: маршруты сообщений

Router сопоставляет команду с функцией-обработчиком. Dispatcher передаёт обработчику зависимости, например путь к базе или объект настроек.

@router.message(Command("add"))
async def handle_add(message: Message, command: CommandObject, db_path: Path):
    if not command.args:
        await message.answer("После /add напишите текст задачи.")
        return

    task_id = add_task(db_path, command.args, user_label)
    await message.answer(f"Задача #{task_id} сохранена ✅")

Почему функция async?

Бот ждёт сеть: Telegram, Google API. Во время ожидания event loop может обрабатывать другие сообщения. Блокирующий gspread вынесен через asyncio.to_thread.

/sync

Только читает и показывает план. Доступен как безопасная диагностика.

/sync_apply confirm

Проверяет числовой ID владельца и требует явное подтверждение перед записью.

ГЛАВА 07

CSV, который понимает Excel

CSV — простой текстовый формат, но совместимость зависит от кодировки, разделителя и опасных начальных символов.

  • UTF-8 BOM
    Помогает старым версиям Excel распознать русский текст.
  • Разделитель
    csv.writer корректно экранирует кавычки и запятые.
  • Formula injection
    Текст, начинающийся с =, +, - или @, нельзя бездумно отдавать Excel как формулу.
  • Память вместо временного файла
    BufferedInputFile отправляет готовые байты напрямую.
Практический тестCSV считается проверенным только после открытия в Excel или LibreOffice. В проекте дополнительно изменялись «Статус» и «Категория».
ГЛАВА 08

Синхронизация без угадывания

Одинаковый id связывает строку Google с записью SQLite. updated_at показывает, какая версия новее.

СитуацияРешение
Задача только в SQLitePUSH_TO_GOOGLE
Задача только в GooglePULL_TO_SQLITE
SQLite обновлена позжеОтправить SQLite-версию
Google обновлён позжеПринять Google-версию
Данные разные, время одинаковоеCONFLICT, ничего не перезаписывать

Почему сначала dry-run?

План отделён от применения. Сначала человек видит количество PUSH, PULL и конфликтов. Только затем отдельная команда запускает запись.

Зачем Apps Script?

Ручное редактирование Google ячейки само по себе не меняет нашу колонку времени. Простой onEdit(e) отслеживает B, E и F и записывает новое UTC-время в G.

Страховочная сеткаПеред применением создаётся копия через sqlite3.backup, а её целостность проверяется PRAGMA integrity_check.
ГЛАВА 09

Доказательства вместо «должно работать»

Тесты не заменяют ручную проверку Telegram и Excel, но позволяют быстро доказать, что новая функция не сломала старые.

python -m unittest discover -s tests -v
python -m py_compile bot.py config.py database.py exporter.py google_sync.py
python -m pip check

Полезная схема диагностики

  1. Прочитать ошибку полностью.
    Путь к Python часто важнее последней строки.
  2. Воспроизвести.
    Одинаково ли падает при повторном запуске?
  3. Локализовать слой.
    Telegram, config, SQLite, CSV или Google?
  4. Исправить одну причину.
    Не менять пять вещей одновременно.
  5. Повторить полный набор тестов.
    Успешный один тест не гарантирует целый проект.

Контрольные вопросы

Почему нельзя хранить токен прямо в bot.py?

Код попадает в архивы и Git. Любой владелец токена может управлять ботом.

Чем dry-run отличается от apply?

Dry-run только строит план. Apply повторно получает актуальные данные и выполняет разрешённые действия.

Что произойдёт при одинаковом updated_at и разных данных?

Будет создан конфликт. Автоматическая запись остановится.

Почему Telegram owner ID надёжнее username?

Username можно изменить, а числовой ID аккаунта остаётся постоянным.
